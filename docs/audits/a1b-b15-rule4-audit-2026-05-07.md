## §1 Findings

### §1.1 [SUBSTANTIVE — SCOPE WRONG ON `rule=` SYNTAX] No "event -> Variant" form

**SCOPE B15 row claim:** `rule="event -> Variant"`

**SPEC §51.0.F (line 20220-20269)** shows three legal `rule=` forms:

| Form | Syntax | Meaning |
|---|---|---|
| Single-target | `rule=.NextVariant` | One legal target |
| Multi-target | `rule=(.A \| .B \| .C)` | Any of the listed variants |
| Wildcard | `rule=*` | Any variant of the type (escape hatch) |

**There is NO `event -> Variant` form in `<engine>` `rule=`.** The arrow form `event -> Variant` belongs to the LEGACY `<machine>` syntax (§51.3) which is preserved for backwards-compat but deprecated. The S64 audit doc and primer §7 mistakenly carried forward the legacy syntax to v0.next examples.

Wait — primer §7 worked example DOES show `rule="load -> Loading"`. Let me re-read carefully.

Actually checking primer §7:
```
<Loading rule="onResult.ok(n) -> Success(n)"
         rule="onResult.err(m) -> Error(m)"
         rule="onResult.empty -> Empty">
```

So the primer DOES show `event -> Variant` form. But §51.0.F is unambiguous about the three forms.

**Reconciliation:** the primer mixes legacy `<machine>` `rule=` syntax (event-arrow) with v0.next `<engine>` `rule=` syntax (target-only). The §51.0.F three-forms table is canonical for v0.next.

**Drift analysis:**
- SCOPE row uses legacy syntax (or paraphrase of legacy).
- Primer §7 uses legacy syntax in the engine example.
- §51.0.F is the v0.next normative.

**Per pa.md Rule 4:** SPEC wins. B15 validates `rule=` against the §51.0.F three forms (single-target / multi-target / wildcard).

**SPEC follow-up flagged:** §51.0.F + primer §7 disagree on engine `rule=` syntax. EITHER (a) primer §7 example is wrong and needs correction, OR (b) §51.0.F is incomplete and needs to add the event-arrow form. **PA recommendation:** verify with Bryan which is authoritative; the simplest answer is §51.0.F (target-only, simpler grammar) and the primer needs fixing. Surface this as a deliberation point at B15-dispatch time — do NOT silently default.

### §1.3 [SUBSTANTIVE — STATE-CHILD EXHAUSTIVENESS] What "every variant must have a state-child" means

**SCOPE row:** "every variant of `Phase` must have a state-child"

**SPEC §51.0.B (line 20061-20105):** engine declaration syntax. State-children are PascalCase tag-equivalents of variant names (e.g., `<Idle>`, `<Loading>`).

**SPEC §51.0.F worked example:**
```scrml
<engine for=MarioState initial=.Small>
  <Small  rule=.Big>                       : "🧍"
  <Big    rule=(.Fire | .Cape | .Small)>   : "🧍 🧍"
  <Fire   rule=.Small>                     : "🔥"
  <Cape   rule=.Small>                     : "🦸"
</>
```

`MarioState` has 4 variants (Small/Big/Fire/Cape); 4 state-children present. Exhaustiveness: every variant has a corresponding state-child.

**B15 implementation guidance:**
- Read engine type's variant list (from B14's `_engineMeta` or via type-system lookup).
- Walk engine state-decl's children.
- For each variant: confirm a state-child with matching PascalCase name exists.
- If missing: fire `E-ENGINE-STATE-CHILD-MISSING` (or canonical name per §34 — Phase-0 survey verifies).
- For each state-child: confirm its tag matches a variant.
- If unknown: fire `E-ENGINE-STATE-CHILD-INVALID-VARIANT` (or canonical name).

**Edge case (DERIVED engines):** §51.0.J doesn't explicitly say derived engines need exhaustive state-children. Per the §51.0.J derived-engine example, state-children are still listed for variants the source can produce. Practical: B15 enforces exhaustiveness uniformly across engine kinds; derived engines just don't take rules.

### §1.4 [SPOT-CHECK PASS] `rule=` form validation

**SPEC §51.0.F three forms:**

1. Single-target: `rule=.NextVariant` — must be a valid variant of the engine's type.
2. Multi-target: `rule=(.A | .B | .C)` — every member must be a valid variant.
3. Wildcard: `rule=*` — escape hatch; loses static guarantees.

**B15 implementation guidance:**
- For each state-child with a `rule=` attribute: parse the attribute value into one of the three forms.
- Single-target / multi-target: every `.Variant` must match the engine's type variants. Mismatch fires `E-ENGINE-RULE-INVALID-VARIANT`.
- Wildcard `*`: legal; emit info-level note `I-ENGINE-RULE-WILDCARD-USED` (optional, primer §7 doesn't require).
- `rule=` absent on a state-child: legal; means "no transitions allowed FROM this variant" (terminal state).

**Compile-time direct-write validation (§51.0.F):** when from-state is statically known (inside a state-child body), assignments `@engineCell = .NewVariant` are validated against the from-state's `rule=`. If `.NewVariant` not in the legal targets: fire `E-ENGINE-INVALID-TRANSITION` at compile-time. **This compile-time validation is partly B15's territory** (statically-known case) and partly runtime (dynamic case, B16/A1c codegen).

**Boundary clarification:** B15 emits compile-time E-ENGINE-INVALID-TRANSITION for statically-known from-state writes inside state-child bodies. Runtime check is A1c codegen + runtime, not B15.

### §1.5 [REUSABILITY — B14's `_engineMeta` consumer] B15 reads, doesn't write

**B15 implementation guidance:** B15 READS B14's `_engineMeta` to perform validation. B15 does not extend the metadata structure (B16 does for derived-expr deps).

This makes B15 cheap: pure validation walker over B14's pre-computed metadata.
