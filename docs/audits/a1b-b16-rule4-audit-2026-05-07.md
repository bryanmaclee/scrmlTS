## §1 Findings

### §1.1 [SUBSTANTIVE — REUSABILITY] B16 fires E-DERIVED-ENGINE-CIRCULAR via B7's generic detectCycle

**SCOPE row** doesn't mention E-DERIVED-ENGINE-CIRCULAR.

**SPEC §51.0.J (line 20403):**
> "Chained derivation (A → B → C) | LEGAL. Cycle detection at compile time → `E-DERIVED-ENGINE-CIRCULAR` (§34)."

**SPEC §31.5 (line 13702-13703):** explicitly cross-refs `E-DERIVED-ENGINE-CIRCULAR` for derived engines.

**B7 SHIPPED state (S67, primer §13.7 B7 specifics):** generic `detectCycle(adj, allNodes)` parameterized on adjacency map + node set. Per primer "B16 (engine-derived, `E-DERIVED-ENGINE-CIRCULAR`) and B10/B11/B12 (validator-arg deps, §31.4) will reuse the same DFS with their own filtered adjacency."

**B16 implementation guidance:**
- For each engine cell with `_cellKind === "engine"` AND `_engineMeta.derivedExpr` present (B14's annotation): walk the derived-expr ExprNode collecting reactive cell reads.
- Emit dep-edges in B7's existing dep-graph (`reads` edges between two `reactive` DG nodes — same shape B7 uses for derived-cell deps).
- New `buildEngineDerivedAdj` filter on the dep-graph (sibling of B7's `buildDerivedReadsAdj`) — restricts edges to engine-cell → upstream connections.
- Run `detectCycle` on this filtered adjacency. Fire `E-DERIVED-ENGINE-CIRCULAR` on cycles.

**Required B16 brief addition:** explicitly enumerate this reuse. B16 is the SECOND consumer of B7's reusability promise (B10 was the first per B10 audit §1.4). Pattern is established.

### §1.2 [SPOT-CHECK PASS] §51.0.J derived-engine rejections enumerated

Per §51.0.J table (line 20393-20405):

| Rule | Behavior | Error |
|---|---|---|
| `derived=expr` | REQUIRED on derived engines | (no rejection — required attr) |
| `rule=` on state-children | REJECTED | `E-DERIVED-ENGINE-NO-RULES` (§34) |
| `initial=` on engine | REJECTED | `E-DERIVED-ENGINE-NO-INITIAL` (§34) |
| Direct writes to auto-declared variable | REJECTED | `E-DERIVED-ENGINE-NO-WRITE` (§34) |
| `<onTransition>` and `effect=` on state-children | LEGAL | (no rejection) |
| Initial-value undefined when source in `initial=` state | (runtime check) | `E-DERIVED-ENGINE-INITIAL-UNDEFINED` (§34) |
| Chained derivation | LEGAL | — |
| Cycle in chains | (compile-time) | `E-DERIVED-ENGINE-CIRCULAR` (§34) |

**B16 implementation guidance:**
- B16 fires: `E-DERIVED-ENGINE-NO-RULES`, `E-DERIVED-ENGINE-NO-INITIAL`, `E-DERIVED-ENGINE-NO-WRITE`, `E-DERIVED-ENGINE-CIRCULAR`.
- B16 does NOT fire: `E-DERIVED-ENGINE-INITIAL-UNDEFINED` (runtime; A1c codegen + runtime emit).

**Walker dispatch on derived engines:**
- For each engine cell with `_cellKind === "engine"`:
  - If `_engineMeta.derivedExpr === null`: not a derived engine; skip B16 checks; B15 handles non-derived rules.
  - If `_engineMeta.derivedExpr !== null`:
    - For each state-child: if `rule=` attribute present → fire `E-DERIVED-ENGINE-NO-RULES`.
    - If `_engineMeta.initialVariant !== null` (i.e., `initial=` was set): fire `E-DERIVED-ENGINE-NO-INITIAL`.
    - For each direct write to the engine's auto-declared variable in source: fire `E-DERIVED-ENGINE-NO-WRITE`. (Walker integrates with the symbol-table walker that handles E-DERIVED-WRITE for const-derived cells — same concept; engine cells are derived-write-rejected too.)

### §1.3 [BOUNDARY CLARIFICATION] E-ENGINE-INVALID-TRANSITION partly B15, partly B16

**SCOPE B16 row** lists `E-ENGINE-INVALID-TRANSITION` as B16 territory.

**SPEC §51.0.F:** E-ENGINE-INVALID-TRANSITION fires on direct writes (compile-time when from-state statically known; runtime otherwise).

**B15 audit §1.4:** said B15 emits compile-time E-ENGINE-INVALID-TRANSITION inside state-child bodies (where from-state is known).

**Drift analysis:** SCOPE row's assignment to B16 is misleading. The compile-time check belongs to whichever walker visits direct-write sites with from-state knowledge. Runtime check is A1c codegen.

**Resolution recommendation:**
- B15 fires compile-time E-ENGINE-INVALID-TRANSITION inside non-derived engine state-child bodies (where from-state statically known per §51.0.F line 20250-20262).
- B16 fires E-DERIVED-ENGINE-NO-WRITE for ALL direct writes to derived engine cells (no compile-time invalid-transition needed; the entire write path is rejected on derived engines).
- A1c codegen + runtime emits runtime E-ENGINE-INVALID-TRANSITION for non-derived engines when from-state is dynamic.

**B16 brief clarification:** B16 owns E-DERIVED-ENGINE-NO-WRITE (derived-engine writes), not the general E-ENGINE-INVALID-TRANSITION. The general invalid-transition is split between B15 (compile-time, statically-known) and A1c (runtime, dynamic).

### §1.5 [.advance VALIDATION] Method-style transition API

**SPEC §51.0.G (line 20270-20294):** `.advance(.X)` is method-style transition; same rule= validation as direct write; throws on invalid; no silent-no-op (`.tryAdvance` rejected).

**B16 implementation guidance:** `.advance` validation is symmetric with direct-write validation per §51.0.F. The walker for direct-write sites also walks `.advance(.X)` calls.

**On derived engines:** `.advance` is REJECTED per §51.0.J (since direct writes are rejected, `.advance` is too). Same `E-DERIVED-ENGINE-NO-WRITE` fires.

**B16 brief addition:** include `.advance(.X)` calls in the walker's direct-write enumeration. AST shape: `MemberCall` with receiver `@engineCell` and method name `advance`.
