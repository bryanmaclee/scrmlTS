## Plan

### W3.1 — bare-null-literal sweep

Extend `gauntlet-phase3-eq-checks.js` to flag bare `null` / `undefined` literal nodes anywhere they appear in value position in the AST — declarations, returns, object property values, array elements, ternary branches, default params, etc.

Detector approach: introduce a second walker pass that visits every exprNode and emits E-SYNTAX-042 when a bare `lit{litType:"null"}` or `lit{litType:"undefined"}` (or unbound `ident{name:"null"|"undefined"}`) is encountered anywhere except as the right-side of an equality comparison (which is already handled by the existing equality detector — must not double-emit).

`<div class="${@x == null ? a : b}">` — the `${...}` segment is preserved as raw text in `kind:"string-literal"` attribute values and never parsed. Default approach: tactical (option b) — localized re-parse of `${...}` segments inside attribute string-literals within the GCP3 detector.

### Diagnosis summary (W3.1)

The W3 detector caught `null` / `undefined` only when the literal was an operand of `==` / `!= /` / `===` / `!==` (via `checkEqNode`). Bare `null` / `undefined` in value position (declaration init, return expr, object property value, array element, ternary branch, function-body assignment RHS) silently passed because no walker visited those positions specifically.

Fix: added `forEachLitNull(node, onLitNull)` that recursively visits every exprNode subtree and emits on `lit{ litType: "null" | "undefined" }` and `ident{ name: "null" | "undefined" }`. To avoid double-emit with `checkEqNode`, suppresses direct lit-null operands of binary `==`/`!=`/`===`/`!==`. To avoid spurious emits on parser desugar, also suppresses direct lit-null operands of binary `is-not` / `is-some` / `is-not-not` (these synthesize a `lit{null}` right operand that is NOT a real source token).

### Diagnosis summary (W3.2)

`<div class="${@x == null ? a : b}">`: the markup tokenizer represents the attribute value as `kind:"string-literal"` with the raw `${...}` preserved as text (no `exprNode`). The W3 detector inspected `attrs[*].value.exprNode` (which is `undefined` for string-literal kind) and never entered the interpolation segments.

Fix (option (b) tactical, default per dispatch): `extractTemplateInterpSegments(raw)` scans the string for `${...}` occurrences with brace-depth tracking. Each segment is parsed via the existing `parseExprToNode` from `expression-parser.ts`. The resulting exprNode is fed back through `inspectExprNode`, which runs both equality detection (W3) and bare-null detection (W3.1). No AST shape change required.
