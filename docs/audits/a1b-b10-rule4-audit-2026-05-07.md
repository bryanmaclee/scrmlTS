# A1b B10 — Rule 4 audit (pre-dispatch)

## §0 Scope

B10 = "Validator type-checking — predicate vocabulary type signatures; arg-type matching; cross-field args resolved via B3's @name" (SCOPE §4.3 row B10, est 5-7h, locks none — but powers B11/B12/B13).

This audit reads §55.1 (universal-core vocabulary), §55.2 (state-cell loci), §55.3 (refinement-type loci), §55.4 (schema loci), §55.11 (cross-field), §55.14 (engine + derived edges), §53 (refinement-type cross-ref), §31.4 (cross-field dep tracking), §34 (error catalog) — and flags drift BEFORE the B10 dispatch encodes the SCOPE wording.

## §1 Findings

### §1.1 [SUBSTANTIVE — CROSS-LOCI EXPANSION] L4 mandates the catalog be reusable across three loci

**SCOPE B10 row:** "predicate vocabulary type signatures; arg-type matching; cross-field args resolved via B3's @name"

**SPEC §55.1 (line 24269-24275, "shared validator core vocabulary (L4)"):**
> "A small fixed set of predicates ('the universal core') is the shared validator vocabulary across all three loci where validation appears: state-cell declarations (§55.2), refinement type expressions (§55.3), and schema columns (§55.4). The same predicates fire with the same semantics at all three; what differs is the enforcement context (compile-time vs runtime, blocking vs reporting), not the validator name."

**Drift analysis:** the SCOPE row's wording is loci-agnostic but doesn't EXPLICITLY require B10's signature catalog to be reusable across all three loci. L4 mandates it.

**Three loci that consume B10's catalog:**

1. **State validators (§55.2)** — the immediate consumer; fires on Shape-2 cells.
2. **Refinement types (§55.3)** — `string(pattern(/.../))`, `number(min(0) && max(100))`. Same predicate names; same arg-type signatures. Different firing semantics (compile-time + runtime boundary).
3. **Schema columns (§55.4)** — `email: text req length(>=2)`. Same predicate names; same arg-type signatures. Different firing semantics (lowering to SQL DDL).

**Required B10 implementation expansion:** the predicate-type-signature catalog SHOULD be a single source of truth, exported as a module, consumed by:
- B10 itself (state validators).
- B21 (refinement-type predicates basic three-zone, §53).
- §39 schema column lowering (already shipped per primer §10 stdlib catalog; B10's catalog might supersede or align).

**Cost impact:** modest expansion — the catalog is the same data structure consumed by different validators. Maybe 30-60min of API design; the underlying catalog rows are the same.

### §1.2 [SUBSTANTIVE — VOCABULARY VERIFICATION] 14 predicates, NOT the historical 18

**Primer §8 (S66 correction):** the universal-core vocabulary is **14 predicates**: `req`, `is some`, `length(<rel-arg>)`, `pattern`, `min`, `max`, `gt`, `lt`, `gte`, `lte`, `eq`, `neq`, `oneOf`, `notIn`.

**NOT in universal-core (per S66 audit at `docs/audits/a1c-roadmap-rule4-audit-2026-05-07.md` §1.1):**
- `email`, `url`, `numeric`, `integer` — these are stdlib `scrml:data` library predicate-builders (separate surface).
- `custom` — this is the `ValidationError` enum tag at §55.9 line 24534, NOT a universal-core predicate.

**B10 implementation guidance:** B10 catalogs the 14 universal-core predicates. Library-surface predicates (`email`, `url`, etc. from `scrml:data`) flow through a SEPARATE registration path — they're user-defined predicate-builders that produce universal-core predicates underneath.

**Required B10 brief addition:** explicitly enumerate the 14; do NOT include `email`, `url`, `numeric`, `integer`, `custom` in the universal-core catalog. (Audit §1.1 of A1c roadmap audit covers the precedent; this is the same correction propagated forward.)

### §1.3 [SUBSTANTIVE — TYPE-SIGNATURE FORMAT] Per-predicate arg-type matrix

**SPEC §55.1 (line 24279-24289, table)** specifies argument shapes informally:

| Predicate | Arg shape (formalized) |
|---|---|
| `req` | (no args) |
| `is some` | (no args; bareword) |
| `length(<rel>)` | RelationalPredicate with numeric threshold; cell type must be string \| array |
| `pattern(re)` | regex literal; cell type must be string |
| `min(n)`, `max(n)` | numeric literal or expression of numeric type; cell type must be number |
| `gt(expr)`, `lt(expr)`, `gte(expr)`, `lte(expr)` | expression comparable to cell type (number \| string \| Date) |
| `eq(expr)`, `neq(expr)` | expression of cell type (any comparable) |
| `oneOf([...])`, `notIn([...])` | array literal of cell type (or compatible enum variants) |

**B10 implementation guidance:** the type-signature catalog encodes:
- Number of args (0, 1, or N).
- Per-arg type (numeric, string, regex, RelationalPredicate, array of cell-type, comparable expression).
- Cell-type compatibility (which cell types accept this predicate).

**E-TYPE-031 family** — applying `pattern(re)` to a `number` cell is a compile-time type error (per §55.1 line 24295). B10 fires this on signature mismatch.

### §1.4 [WAVE-ORDERING CORRECTION + REUSABILITY] B10 fires E-VALIDATOR-CIRCULAR-DEP via B7 reuse

**SPEC §55.11 line 24631:**
> "Circular deps | `<a eq(@b)>` + `<b eq(@a)>` → `E-VALIDATOR-CIRCULAR-DEP` (§34) at compile time. The validator-dep graph is a DAG; cycles are forbidden."

**SPEC §31.4** — cross-field validation predicate-arg deps use "the same machinery as the standard reactive dependency tracker." Cross-ref §6.6 explicitly.

**B7 SHIPPED (S67):** generic `detectCycle(adj, allNodes)` + reusable dep-edge structure (per primer §13.7 B7 specifics — "B10/B11/B12 will reuse the same DFS").

**B10 implementation guidance:**
- For each validator with cross-field args (`@cell` references in arg expressions), emit dep-edges into B7's existing dep-graph.
- Use `buildValidatorArgsAdj` (or similar) — a sibling of `buildDerivedReadsAdj` — to filter edges by validator-arg origin.
- Run `detectCycle` on this filtered adjacency.
- Fire `E-VALIDATOR-CIRCULAR-DEP` on cycles.

**Required B10 brief addition:** B10 fires `E-VALIDATOR-CIRCULAR-DEP` (§34, §55.11) using B7's generic cycle detector. This is the FIRST consumer of the B7 reusability promise; lays the pattern for B11/B12/B16.

### §1.5 [REUSABILITY — B3 INTEGRATION] Cross-field args resolved via B3's @name

**SCOPE row:** "cross-field args resolved via B3's @name"

**B3 SHIPPED state (primer §13.7 B3 specifics):** every `@`-prefixed `IdentExpr` reachable via SYM PASS-3 carries `_resolvedStateCell: StateCellRecord | null | undefined`. `getResolvedStateCell(ident)` is the public read API.

**B10 implementation guidance:**
- When walking validator-arg ExprNodes (output of B9), every `@cell` reference is an IdentExpr with B3's `_resolvedStateCell` already set.
- Resolution failure (`_resolvedStateCell === null`) — B10 fires E-SCOPE-001 or equivalent unresolved-cell error.
- Compound-nav (`@signup.password`) — B3 resolves the BASE cell; B10 must re-resolve the leaf via `lookupQualifiedStateCell` (per primer §13.7 B3 specifics on compound nav).

**Cost impact:** modest. B3 did the heavy lifting; B10 reads annotations.

### §1.6 [BOUNDARY — DERIVED-CELL REJECTION IS B13, NOT B10] §55.14 routes to B13

**SPEC §55.14 line 24679-24681:**
> "Validators on derived cells (`const <x ...>` or `<engine derived=>`): REJECTED. Per open-Q §7.9 resolution: derived cells are READ-ONLY; validators imply gating which is incoherent on a read-only computed value. The compiler emits `E-DERIVED-WITH-VALIDATORS` (§34) at parse-time."

**SCOPE row B13:** "**E-DERIVED-WITH-VALIDATORS** + 4-level error-message resolution-chain recording — derived cells reject validator attrs (Step 5 left them as syntactic but A1b rejects)..."

**Drift analysis:** none. B10 type-checks validators; B13 fires the rejection on derived cells. B10 should NOT pre-empt B13's territory — type-check the args (B10) and let B13 own the "is this allowed at all" decision.

**B10 implementation guidance:** type-check validator args UNIFORMLY across all cells that have validators (including derived cells). The "validators on derived cells are rejected" is a separate B13 walker pass. If B10 fires no error on a derived cell with validators (because B10 only type-checks), B13's pass picks it up and fires E-DERIVED-WITH-VALIDATORS.

This separation keeps responsibilities clean and avoids B10/B13 overlap.

### §1.7 [BOUNDARY — ENGINE STATE-CELL VALIDATORS] §55.14 says legal-but-typically-redundant

**SPEC §55.14 line 24669-24677:**
> "Validators on engine auto-declared variables: ... universal-core vocabulary applies (e.g., `oneOf([...])` makes sense). HOWEVER, the engine's `rule=` contract is the canonical mechanism... Validators on engine cells are LEGAL but typically REDUNDANT when `rule=` is the constraining contract. When both apply (rule= + a validator), both fire; the cell is valid only if both agree."

**B10 implementation guidance:** type-check validators on engine cells normally. No special rejection. Both rule= and validator must agree at runtime — that's an A1c codegen / runtime concern, not B10's.

### §1.8 [REUSABILITY] B10's catalog feeds B11/B12/B16

The integration story:

```
B10 (this audit): predicate type-signature catalog + per-arg type matching.
                   Fires E-VALIDATOR-CIRCULAR-DEP via B7 reuse.
                   Resolves cross-field args via B3.

B11 (audited):    consumes B10's catalog for synth-cell reactive wiring
                   (cross-field deps emit dep-edges via B7).

B12 (audited):    extends B11 to per-field; same B10 catalog reuse.

B13 (audited):    fires E-DERIVED-WITH-VALIDATORS on derived cells with
                   validator attrs. Independent of B10's type-checking.

B16 (future):     engine-derived expressions may consume universal-core
                   predicates (§55.14 implicitly via rule= validator
                   coexistence). B10 catalog reusable.

B21 (future):     refinement-type predicates (§53). Same vocabulary,
                   different firing semantics. B10 catalog reusable.
```

This integration is sound. B10's catalog is the single source of truth for predicate type signatures across the compiler.

## §2 B10 dispatch brief — required additions beyond SCOPE row

When PA writes the B10 dispatch brief, the following MUST be in the brief:

1. **Predicate-type-signature catalog is a single source of truth, reusable across loci** (state validators, refinement types, schema columns per L4). Export as a module so B11/B12/B16/B21 reuse without duplication.

2. **14 universal-core predicates per §55.1**, NOT the historical 18 (no `email`, `url`, `numeric`, `integer`, `custom`). Library-surface predicates (`scrml:data`) flow through separate registration; they BUILD on universal-core.

3. **Per-predicate type signature** specifies: arg count, per-arg type (numeric/string/regex/RelationalPredicate/array-of-cell-type/comparable-expression), cell-type compatibility.

4. **Fires `E-TYPE-031` family** (signature mismatch — applying `pattern(re)` to a `number` cell, etc.) per §55.1 line 24295.

5. **Fires `E-VALIDATOR-CIRCULAR-DEP`** (§55.11, §34) via B7's generic `detectCycle` + a new `buildValidatorArgsAdj` filter. This is the FIRST consumer of B7's reusability promise — establishes the pattern for B11/B12/B16.

6. **Cross-field `@cell` references resolved via B3's `_resolvedStateCell` annotation** (per primer §13.7 B3 specifics). Compound-nav (`@signup.password`) requires `lookupQualifiedStateCell` for leaf resolution.

7. **Derived-cell rejection is B13's job, NOT B10.** B10 type-checks args uniformly; B13 owns "validators forbidden on derived cells."

8. **Engine state-cell validators are legal** (§55.14); type-check normally. Coexistence with `rule=` is runtime-level concern, not B10's.

9. **Phase-0 survey gate:** confirm (a) §55.1 vocabulary count is 14 (corroborate with primer §8 + A1c roadmap audit §1.1), (b) B7's `detectCycle` API is callable from a sibling-of-`runDG` location (or B10 hooks into runDG itself), (c) B3's `_resolvedStateCell` annotation is reliable on @cell refs inside validator-arg ExprNodes (post-B9), (d) the existing schema-column lowering (per primer §10 §39) doesn't already encode a competing predicate-signature catalog — if it does, reconcile.
