## §1 Findings

### §1.1 [SUBSTANTIVE — SCOPE UNDERSPECIFIES] B17 must validate `effect=` AND `<onTransition>` placement

**SCOPE B17 row** mentions only `<onTransition>` element + `E-COMPONENT-ENGINE-SCOPE`.

**SPEC §51.0.H (line 20298-20347)** distinguishes TWO surface forms:

1. **`effect=` attribute** on a state-child (single-target only).
2. **`<onTransition>` element child** of a state-child (multi-target or attribute-bearing).

**Both have placement constraints + form constraints:**

| Form | Constraint | Error |
|---|---|---|
| `effect=` | Single-target `rule=` only; multi-target → ambiguous | `E-ENGINE-EFFECT-AMBIGUOUS` (§34) |
| `effect=` | Engine-only (forbidden in `<match>` blocks) | (§18.0.2 cross-ref) |
| `<onTransition>` | Engine state-child only (forbidden in `<match>` blocks) | (§18.0.2 cross-ref) |
| `<onTransition>` | Required attribute: `to=` OR `from=` (one of) | (Phase-0 survey verifies §34 naming) |
| `<onTransition>` | `once`, `if=expr` are optional bare/expr attrs | — |

**Drift analysis:** SCOPE row underspecifies. B17 must validate both forms.

**Required B17 implementation expansion:**
- Walk every engine state-child (post-B14 annotation).
- For each `effect=` attribute: validate single-target `rule=` invariant. Fire `E-ENGINE-EFFECT-AMBIGUOUS` if `rule=` is multi-target or wildcard.
- For each `<onTransition>` child element: validate placement (only inside engine state-children, not match arms). Fire `E-STRUCTURAL-ELEMENT-MISPLACED` (per §4 + §24 structural-elements registry, primer §9.6) if found in a `<match>` arm or other invalid context.
- For each `<onTransition>` element: validate required `to=` OR `from=` attribute. Fire `E-ONTRANSITION-NO-DIRECTION` (or canonical name per §34 — Phase-0 survey).

**Cost impact:** ~30-60min added to B17 for the `effect=` validation. Modest.

### §1.2 [SUBSTANTIVE — WAVE-4 CLOSER FIRE-SITE] E-COMPONENT-ENGINE-SCOPE owned by B17 BUT fired by B14

**SCOPE B17 row:** owns E-COMPONENT-ENGINE-SCOPE.

**B14 audit §1.5:** recommended B14 fires E-COMPONENT-ENGINE-SCOPE at the engine-decl walk site (when surrounded by component body). Avoids two-walker churn.

**Reconciliation:** "ownership" is conceptual; "fire site" is the walker location. B17 OWNS the scope (the broader components-vs-engines distinction territory) but the FIRE SITE for engine-inside-component is B14's walker.

**B17 implementation guidance:**
- B17 may fire E-COMPONENT-ENGINE-SCOPE for OTHER cases of components-vs-engines violations (e.g., `<EngineName/>` mount tag inside a component body — different from engine-decl inside component body).
- The "engine declaration inside component body" fire site is B14's per audit §1.5.
- B17 covers the residual cases.

**Phase-0 survey item:** enumerate all components-vs-engines cases:
- Engine declaration inside component body: B14 fires.
- Engine mount tag (`<EngineName/>`) inside component body: B17 fires (?).
- Component declaration inside engine body: legal per §51.0.K line 20428 ("an engine body MAY instantiate components"). No fire.

### §1.3 [SPOT-CHECK PASS] §51.0.H attribute semantics

**Built-in `<onTransition>` attributes** per §51.0.H line 20325-20332:

| Attribute | Meaning | B17 validation |
|---|---|---|
| `to=.Variant` | Fires when LEAVING from-state TOWARD `.Variant` | Must be a valid variant of engine type (similar to rule= variant validation per B15). |
| `from=.Variant` | Fires when ARRIVING at to-state FROM `.Variant` | Must be a valid variant. Placed in TARGET state-child. |
| `once` | Bare attr — handler runs at most ONCE | (no value validation; presence-only) |
| `if=expr` | Conditional gating | (expression typing happens later; not B17's territory) |

**B17 implementation guidance:** validate `to=` / `from=` variant references against the engine's variant list (B14's `_engineMeta`). Fire `E-ONTRANSITION-INVALID-VARIANT` (canonical name TBD; Phase-0 survey).

**Edge case:** `<onTransition>` with BOTH `to=` and `from=`. §51.0.H doesn't explicitly forbid. Spec ambiguous. **Phase-0 survey item:** verify intended behavior (likely both-forbidden — they're alternative directionalities).

### §1.4 [BOUNDARY] §18.0.2 — match-vs-engine effect/onTransition restriction

**SPEC §18.0.2 (cross-ref from §51.0.H line 20346-20347):**
> "`effect=` and `<onTransition>` are FORBIDDEN inside `<match>` blocks (Tier 1 is read-only). They are engine-only."

**B17 implementation guidance:**
- Walk markup tree.
- For each `<match>` block: descend into its arms. If any arm contains `effect=` attribute or `<onTransition>` element → fire `E-STRUCTURAL-ELEMENT-MISPLACED` (per primer §9.6 structural-elements registry — "scrml-defined structural elements (NOT HTML). E-STRUCTURAL-ELEMENT-MISPLACED if used in unsupported contexts.").

**Reuse:** `E-STRUCTURAL-ELEMENT-MISPLACED` is the canonical "scrml-special element used in wrong context" error. B17 reuses; doesn't introduce a new code.

## §2 B17 dispatch brief — required additions beyond SCOPE row

When PA writes the B17 dispatch brief, the following MUST be in the brief:

1. **B17 validates BOTH `effect=` AND `<onTransition>` placement + form** (not just onTransition per SCOPE wording).

2. **`effect=` single-target invariant** — fires E-ENGINE-EFFECT-AMBIGUOUS on multi-target or wildcard `rule=`.

3. **`<onTransition>` placement** — engine state-child only. Fire E-STRUCTURAL-ELEMENT-MISPLACED on misuse (e.g., inside `<match>` arms per §18.0.2).

4. **`<onTransition>` direction attributes** — `to=` OR `from=` required (one of). Both-attributes case: Phase-0 survey verifies intended behavior. Fire E-ONTRANSITION-NO-DIRECTION on absence; E-ONTRANSITION-INVALID-VARIANT on unknown variant.

5. **`once`, `if=expr` attributes** — `once` is presence-only (no value validation in B17); `if=expr` expression-typing is not B17's territory.

6. **E-COMPONENT-ENGINE-SCOPE fire-site is B14 (engine-decl-inside-component)**; B17 owns RESIDUAL cases (e.g., engine mount tag inside component body — Phase-0 survey enumerates).

7. **Reuse E-STRUCTURAL-ELEMENT-MISPLACED** (per primer §9.6) for misplaced engine-special elements; do NOT introduce a new code.

8. **Phase-0 survey gate:** confirm (a) §34 catalog rows for E-ENGINE-EFFECT-AMBIGUOUS, E-ONTRANSITION-NO-DIRECTION, E-ONTRANSITION-INVALID-VARIANT (canonical naming), (b) `<onTransition>` both-direction-attributes case behavior, (c) component-vs-engine cases enumerated for B14-vs-B17 fire-site assignment.
