# P5-14 deferral — single-locus fix has Dropdown-shaped regression

## Verdict

**P5-14 cannot be closed as a single-locus tag-frame.js fix.** The brief's
diagnosis is structurally CORRECT for the three target files (closeTagFrame
must pop on mismatch — mirrors live BS `popTagContext` at
block-splitter.js L1576-1586). But the simple pop has a non-trivial ripple
through parse-expr.js's `parseMarkupValue` slice path that REGRESSES
`samples/compilation-tests/gauntlet-r10-ts-components.scrml` from EXACT
into DIFF-hoist-count (loses the `Dropdown` component-def).

The fix needs to thread a "slice-mode" flag through `parseMarkupTrace` so
the in-slice mismatch can BAIL (matching the live parseLiftTag) while the
file-level mismatch can POP (matching the live BS). That is a 3-file
change — beyond the brief's "single-locus" scope.

## Root cause of the regression

`samples/compilation-tests/gauntlet-r10-ts-components.scrml` defines four
inline `const ComponentName = <markup>` component-defs inside `${...}`
logic blocks. The Dropdown body contains a structural redundancy
(arguably a corpus bug — the author wrote both `</>` AND `</button>`):

```scrml
// lines 160-162
<button class="dropdown-trigger" onclick=toggleOpen()>
    ${value ? value : placeholder ?? "Select..."}</>
</button>
```

The `</>` cleanly closes `<button>`; the following `</button>` is then
an EXPLICIT closer against the OUTER `<div class="dropdown">` (mismatch:
`</button>` vs `<div>`).

The LIVE pipeline handles this fine because it has TWO different
mismatch-recovery code paths:
- **BS layer** (block-splitter.js L1576-1586) — POP the open frame.
- **parseLiftTag** (ast-builder.js L3098-3099) — BAIL (return null), the
  caller resets cursor and falls through to expression-RHS / string init
  with raw-text component-def detection at ast-builder.js L8048-8056.

The Dropdown body is parsed by parseLiftTag (called from the const-decl
RHS path inside `${...}`), so the BAIL semantics applies. The const init
becomes a STRING; live's component-def detector sees a string starting
with `<` and creates the component-def via the raw-text path.

The NATIVE pipeline has ONE mismatch path (closeTagFrame). With the
simple pop fix:
- The slice (`<div class="dropdown">...</button>...</div>`) is parsed by
  `parse-expr.js`'s `parseMarkupValue` via a fresh `parseMarkupTrace`
  invocation on the substring.
- The slice's `parseMarkup` ctx has an EMPTY `blockContextStack` — any
  context-based gate (e.g., "skip pop if `topBlockContextFrame !== null`")
  fails to discriminate this case.
- The `</button>` mismatch pops the slice's ROOT `<div>` prematurely.
- `parseMarkupValue` returns a TRUNCATED MarkupValue (the `<div>` element
  only contains the `<button>` subtree, not the trailing
  `<div class="dropdown-menu">...</div>`).
- The const-decl's init is technically still a MarkupValue, but the
  truncation cascades downstream — the JS-layer cursor is advanced to
  mid-content, statement-parsing errors (E-STMT-CLASS-NAME / E-IF /
  E-EXPR-UNEXPECTED) hit the trailing `<div ...>...</div></div>}` bytes.
- The const-Dropdown is no longer recognized as a VarDecl with a
  MarkupValue init (the stmt-parse fails before the VarDecl materializes),
  so `collect-hoisted.js`'s `collectComponentDefs` does not emit it.

Result: 1 component lost; `hoist[components live=4 native=3]`.

## Why the obvious context-aware gate fails

The intuitive gate "pop only at top-level markup (outside any braced
context)" fails because the slice is parsed in its OWN parseMarkup
context with no enclosing braced frames — `topBlockContextFrame(ctx) ===
null` is TRUE both for the file-level top and for the slice. The two
caller intents are indistinguishable from inside parseMarkup.

The slice opens with `<` at slice-local byte 0 — a "popped frame's
openSpan starts at 0" heuristic could discriminate, but breaks the 16
corpus files that legitimately start with `<` at byte 0 (no leading
comment).

## Recommended P5-14 v2 — sub-units

**Option A (preferred — minimal scope expansion):** Thread a
`{ inMarkupValueSlice: true }` option through `parseMarkupTrace` →
`runMarkup` → `ctx.inMarkupValueSlice`. `handleCloser` reads this flag
and passes `{ allowMismatchPop: !ctx.inMarkupValueSlice }` to
`closeTagFrame`. The slice-mode parseMarkup keeps the no-pop bail
semantics; the file-level parseMarkup gets the pop-on-mismatch recovery.

Touch list:
- `compiler/native-parser/tag-frame.js` — accept third `options` arg
  with `allowMismatchPop` (default false → backward-compatible).
- `compiler/native-parser/parse-markup.js` — `runMarkup` reads
  `options.inMarkupValueSlice` and stores on ctx; `handleCloser` derives
  `allowMismatchPop` from `!ctx.inMarkupValueSlice`. `parseMarkupTrace` /
  `parseMarkup` accept the options.
- `compiler/native-parser/parse-expr.js` —
  `parseMarkupViaLazyRequire` passes `{ inMarkupValueSlice: true }` to
  the lazy-required `parseMarkupTrace`.

3 files, additive API, no behavior change for callers that don't pass
the flag. Closes the 3 target files (EXACT +3), no Dropdown regression
(slice-mode stays no-pop), pre-existing unit tests at
`parser-conformance-markup.test.js` L2084-2092 still pass (they call
`closeTagFrame(ctx, closer)` without the options arg, so `allowMismatchPop`
defaults to false — current behavior preserved). The conformance test
comment ("does NOT pop") needs no edit because the default is no-pop;
new test cases would cover the `allowMismatchPop: true` branch.

**Option B (lower scope, partial close):** Land the simple pop, accept
the Dropdown regression, fix the corpus file (`</>` `</button>` →
`</>` only or `</button>` only) so the regression disappears. This is
corpus-cleanup, not parser work — could be a sibling unit. Pros: keeps
P5-14 itself single-locus. Cons: doesn't actually harden the parser
against the live's bail-vs-pop divergence; future similarly-shaped corpus
files would hit the same trap.

**Option C (alternate single-locus):** Look for a heuristic that
distinguishes slice from file inside parseMarkup without an explicit
flag. The CWD investigation found none robust enough — every candidate
(BlockContext stack emptiness, opener-at-byte-0, depth-drains-to-empty,
unwind-to-matching) misclassifies at least one of the four cases.

Recommend **Option A** for the next dispatch.
