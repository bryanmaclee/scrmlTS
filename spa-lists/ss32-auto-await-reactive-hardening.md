1. **g-auto-await-reactive-server-no-error-arm** (MED) `[status=landed-on-branch SHA=d39f9d93]`
   - Symptom: the per-statement auto-await IIFE is **catch-less** → a failed reactive-server assignment silently drops (browser `unhandledrejection`, no scrml error surface). flogence S15 residual.
   - Footprint: give the IIFE an error arm — route the rejection through the statement's `!{}` handler if present, else a scrml-level default error (NOT a silent drop). Mirror the §5.2.2 / CPS error-routing the rest of the pipeline uses.
   - **LANDED ss32** (`d39f9d93`): appended a uniform `.catch(_scrml_async_err => _scrml_error_boundary_log(<name>, _scrml_async_err))` to the wrapped IIFE call (`emit-client.ts` post-server-fn-iife-wrap). R26-verified: before = `unhandledRejection` fired; after = surfaced as a scrml error, no `unhandledRejection`. Chose `_scrml_error_boundary_log` over the briefed `_scrml_error_boundary_uncaught` (the latter *returns* an Error meant to be `throw`n → would re-create the very unhandledrejection). Preserves the S84 stmt-vs-expr `;)` logic; valid in both positions. Hook gate 18061/0, parser-conformance 4276/0, target test 16/0. **DISCOVERED FORK (g) — escalated to PA:** `@cell = serverFn() !{...}` reactive-server assignments emit a DEAD `!{}` dispatch (wrap captures the Promise → `if(result.__scrml_error)` checks a Promise → never fires). Pre-existing, not a regression; the `.catch` safety-nets it but doesn't restore the user's handler. Same structural root as item 2.

2. **g-auto-await-read-before-resolve-race** (LOW, **survey-first**) `[status=parked — REPRODUCES, design-ruling required → PA]`
   - Symptom: a data-DEPENDENT statement after a reactive-server assignment reads the cell synchronously BEFORE the await resolves (flogence-triage tangential, flag-only — no witnessed repro yet).
   - **STOP-first:** reproduce on real source before building (S138 reverse). If it doesn't reproduce → file NOT-REPRODUCED + close, don't dispatch a fix. If it does → the per-stmt auto-await ordering needs a data-dependency barrier (the ss22 mechanism already isolates data-independent stmts; this is the dependent-read corner).
   - **SURVEYED ss32 → REPRODUCES (narrowly), PARKED for PA design ruling.** Witnessed (R26, real compiled source @ `c193797b`): top-level `@derived = @data.field` off an async-server cell emits a **bare one-shot `_scrml_reactive_set`** (NOT wrapped in `_scrml_effect`) → reads `@data` pre-resolve. Two failure modes: (1) no initializer → synchronous `_scrml_reactive_get("data").value` reads `undefined` → **`TypeError` at module-init, whole page init breaks** (item-1's `.catch` does NOT cover this — it's on the async IIFE, not the sync successor); (2) with initializer → `@derived` snapshots the initial value and **stays stale forever** (0 subscriptions on `@data`). **Display bindings `${@data.value}` ARE race-free** (wrapped in `_scrml_effect` → subscribe → re-derive). Survey doc: `docs/changes/g-auto-await-read-before-resolve-race/SURVEY.md`. **Why parked (not dispatched):** the fix is a designer-axis call — (1) narrow dependency-barrier/await-chaining, (2) language-wide reactive-derivation semantics, (3) hybrid async-taint barrier, OR (4) a lint/diagnostic stance ("derive at the read site, not a snapshot cell"). sPA does not make design rulings. **Shares the structural root with item-1 fork (g)** — both are "sync successor reads async-deferred producer pre-resolve"; a single barrier could subsume both. PA owns the ruling + any fix dispatch.
